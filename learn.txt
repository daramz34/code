## Mistake 1: Confusing Path vs Query

| Wrong | Why Bad | Right |
|-------|---------|-------|
| `/user?id=42` | Query for required ID | `/user/42` |
| `/products/laptop` | Path for filter | `/products?q=laptop` |

**Rule:** Required identifier = Path. Optional filter = Query.

---

## Mistake 2: Required Query Parameter

| Wrong | Problem | Right |
|-------|---------|-------|
| `q: str` | Must provide or error | `q: Optional[str] = None` |

**Fix:** Add `Optional` and default `= None`.

---

## Mistake 3: Wrong List Index vs ID

| Wrong | Crashes/Wrong | Right |
|-------|-------------|-------|
| `return products[id]` | Index 2 ≠ ID 2 | Loop: `if p["id"] == id` |

**Fix:**
```python
for p in products:
    if p["id"] == product_id:
        return p
```

---

## Mistake 4: Modifying List While Iterating

| Wrong | Problem |
|-------|---------|
| `for p in results: if bad: results.remove(p)` | Skips items, bugs |

**Fix:** Create new list.
```python
results = [p for p in results if p["price"] <= max_price]
```

---

## Mistake 5: Case-Sensitive Search

| Wrong | Misses matches |
|-------|----------------|
| `q in p["name"]` | "Laptop" won't match "laptop" |

**Fix:**
```python
q.lower() in p["name"].lower()
```

---

## Mistake 6: Trailing Slash Confusion

| Define | Visit | Result |
|--------|-------|--------|
| `/items/` | `/items` | ❌ 404 |
| `/items` | `/items/` | ❌ 404 |

**Fix:** Pick one, use consistently. I recommend `/` at end.

---

## Mistake 7: Wrong Validation Import

| Wrong | Error |
|-------|-------|
| `from fastapi import path` | `path` not found |

**Fix:**
```python
from fastapi import Path, Query  # Capital P, Q
```

---

## Mistake 8: Mutable Default Arguments

| Wrong | Bug |
|-------|-----|
| `def f(results=[]):` | List shared between calls! |

**Fix:**
```python
def f(results=None):
    if results is None:
        results = []
```

In Computer Science, async (short for asynchronous) is a programming pattern that allows a computer to handle multiple tasks at once without getting "stuck" waiting for one to finish.

Think of it like a Chef in a kitchen versus a Vending Machine.

🍳 The Analogy: Synchronous vs. Asynchronous
Synchronous (The Vending Machine): You put your money in, and you must stand there and wait for the snack to drop before you can do anything else. If the machine is slow, you are stuck. This is "Blocking."

Asynchronous (The Chef): The chef puts a steak on the grill. Instead of staring at the steak for 10 minutes until it's done, the chef turns around and starts chopping vegetables. When the steak is ready, a timer goes off (an "event"), and the chef returns to it. This is "Non-blocking."

🛠️ Why we use it in FastAPI
When you build an API, your code often has to wait for:

The Database to find a record.

A File to be saved to the disk.

Another API to send back data.

Without async, if 100 people try to use your app at the same time and the database is slow, the 101st person has to wait in a "line." With async, your FastAPI server says: "I'm waiting for the database for User A, but I have a split second to help User B while I wait!"

🐍 The Python Keywords: async and await
In Python, these two work as a team:

async def: This tells Python, "This function is a 'coroutine.' It’s allowed to pause and let other code run."

await: This is the "Pause Button." You put it in front of a slow task (like a database call). It tells Python: "Go ahead and do other work; come back to me when this specific task is finished."

Python
@app.get("/items/")
async def get_items(db: Session = Depends(get_db)):
    # The 'await' lets the server handle other users 
    # while the database searches for the items.
    items = await db.execute(query) 
    return items

🐍 Daily "No-Rust" Python Practice: Concurrency
Challenge: Does async mean the computer is doing two things at the exact same millisecond (like a 16-core CPU), or is it just switching between tasks very fast?

A: Doing two things at the exact same time (Parallelism).

B: Switching tasks effectively during "wait times" (Concurrency).

Answer: B! Async is like a waiter in a restaurant. They can't physically carry two trays for two different tables at the exact same time, but they are "concurrent" because they take an order at Table 1, then immediately go refill water at Table 2 while Table 1's food is cooking.




1. What is Pagination?
Imagine you are scrolling through Instagram. You don't download every post ever made since 2010 all at once; your phone would explode. Instead, you load the first 20. When you reach the bottom, you load the next 20.

The "Limit": This is your Page Size. It’s the maximum number of items you want to see in one go.

The "Offset" (or Skip): This is how many items you want to jump over. If your limit is 10 and you want page 2, you tell the database: "Skip the first 10, and give me the next 10."

2. What is Filtering?
This is the "Search" functionality. Instead of getting everything, you only want items that meet a specific condition—like "Items cheaper than #5000" or "Items in the Electronics category." In SQL, this uses the WHERE clause.


\* **Definition:** This is **Conditional Logic**. 
* **Why it's there:** We check if the user actually sent a price. If they didn't, we skip this step. If they did, we "attach" a `WHERE` clause to our SQL command. This is how you implement the **Filtering** objective of your challenge.

### 4. Slicing the Data (The Pagination)
**New Addition:** `.offset(skip).limit(limit).all()`

* **Offset (`skip`):** This tells the database, "Start reading from row X." If `skip` is 5, the database ignores the first 5 items.
* **Limit:** This tells the database, "Stop after you've given me Y items."
* **`.all()`:** This is the **executioner**. Up until this word, no data has been pulled. `.all()` tells SQLAlchemy: "Okay, the list is finished. Go to the database now and bring me back the results as a Python list."
